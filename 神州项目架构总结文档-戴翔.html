业务线概述

神州专车

采用“专业车辆，专业司机”的B2Cyunying提供模式，车辆和司机都由神州提供，定位于中高端群体，主打中高端商务用车服务市场

神州租车

提供汽车租赁，多种类型（长租(年月)/短租(天)/分时租赁(小时)）

汽车金融

为客户提供包括二手车金融、汽车抵押借款等在内的全面、专业、安全、快捷的汽车金融服务。用户有资金需求时，可至线下门店进行咨询，由专业人员对目标车辆进行免费检查及估值，验车通过即可办理相关手续。自由选择分期（3/12/24/36期）

汽车电商

二手车和新车，线下地区建立门店的方式

神州车管家

类似汽车生活和服务的一个社区，面向有车和无车一族（充电加油、道路救援、汽车维修、汽车保险、违章代办等服务）

整体流程

开发流程

1. 需求宣讲

需求宣讲在需求评审完成后，由项目经理组织产品进行宣讲。需求宣讲阶段开发人员工作如下：
1. 宣讲过程中开发人员需根据当前系统情况对业务需求进行评估，对系统性能有影响的处理方式、处理方式不合理的需求以及需求有歧义的部分应及时提出问题并讨论解决方案；
2. 开发人员发现需求中涉及其他团队任务时，应及时确认相关团队是否在场，及时沟通处理方案

2. 需求分析

需求宣讲完成后，开发负责人应对需求进行功能分解，评估各子功能的开发时间和对应开发人员，并完善JIRA信息。JIRA待完善信息如下：
1. 在功能项目中新建主开发JIRA，并填写如下信息：

 - 主题为“需求统筹名称--项目名”，例如：XXXX功能--CRM
 - 问题类型为“任务” 
 - 根据需求分析结果填写“开始时间”、“结束时间”和“工作量”  
 - 测试负责人为“luoyan01” 
 - 其他字段信息根据实际情况填写 
 - 功能JIRA创建完成后将该JIRA relates to 到对应的需求统筹JIRA中

2. 在主开发JIRA下根据功能分解创建子任务，子任务需完善信息如下：

 - 主题为“功能分解的名称”
 - 经办人为实际开发人员  
 - 根据需求分析结果填写“开始时间”、“结束时间”和“工作量” 
 - 测试负责人为“luoyan01” 
 - 其他字段信息根据实际情况填写

3. 需求开发

  1. 开发人员根据各自分配的开发任务进行功能开发，开发过程中如果不能按照JIRA中评估的时间完成开发工作的，需要备注原因并及时调整结束时间。
  2. 开发过程中如果涉及其他开发团队配合的工作应及时确认相关联系人并确认完成时间，如果不能按预期完成开发任务的需要在JIRA中备注原因并及时调整开始和结束时间。
  3. 开发任务开始前需点击“开始进行”，开发完成后需点击“解决问题”，提测后需要点击“关闭问题”
  4. 开发过程中涉及SQL脚本执行可参照“测试&预生产环境SQL脚本执行流程”详细说明

4. 开发自测

开发人员需根据测试人员提供的测试用例，挑选自测用例进行自测，提交测试前必须保证自测通过。

5. 功能提测

基础服务与中间件

下列服务平台用于支撑上层业务系统的平稳运行。查看汇总文档
可前往神州架构平台查看详细

配置中心 (conf-center)

为解决系统运行中配置数据实时的变化，提供数据订阅功能，统一配置(多应用统一管理)
开源框架：disconf

存储中心 (cache-cloud)

提供对cache的资源分配、监控、警报、命令执行（中间件：redis、memcached）

注册中心 (zkcloud)

ZooKeeper服务的统计、监控、报警管理以及自动化运维的平台（zkdoctor优化版的zkcloud）

注册中心的应用场景
常见框架：ZooKeeper、ected
提供统一命名服务，将机器信息，服务注册到注册中心中，由注册中心本身的高可用，一致性为其他应用提供集群支撑

服务治理平台 (prism)

prism是对资源调用的控制组件，主要涵盖对服务的授权(限权)、限流、降级、路由、服务监控等功能(核心依赖于封装的自研服务化框架)
服务化框架包含治理平台，并且完成rpc，弹性横向扩展，优雅停机、服务自动发现、负载均衡、集群容错，多协议（自定义协议、http）等

神州服务化框架
常见框架：dubbo|dubbox、spring cloud（熔断）
与微服务框架配套的平台，如果没有此功能则只是完成rpc部分(比如：hessian、thrift)，当服务变得越来越多的时候，需要有一个平台能管理这些服务
角色：注册中心(registry)、服务消费者（consumer）、服务提供者（provider）
服务存储方式(与dubbo的存储方式一致)
服务启动流程

  1. 注册中心启动（zookeeper）
  2. 服务提供者启动将本机ip等信息注册到注册中心（startInit.properties），并与注册中心保持长连接（心跳检测服务状态）
  3. 服务消费者启动从注册中心拉取对应的服务，并与注册中心保持长连接（订阅服务消息）

startInit.properties

属性 用途
dev 开发阶段 sql 动态加载，修改ibatis的脚本后不用重启即可生效
startRegisterCenter 注册中心启动，注册服务
remoteServiceStart 启动加载远程服务
startNettyServer 启动Netty做服务，通过读取nettyConfig中的配置（包括端口，编码解码handler等）
startCacheModelConfig ibatis自带的缓存机制需要手动配置缓存触发点，而且后期要不断的根据sql的添加去维护触发点，最重要的是在触发缓存清空后，此cacheModel下的所有缓存都将消失。鉴于其以上两个原因，开发了rhs缓存。rhs缓存主要是基于redis和habse实现的，在redis中存放需要缓存的信息，在hbase中根据配置信息自动记录缓存触发点，通过解析binlog获取某数据变化的行信息，到hbase中查询出与该行相关的缓存key，从而将缓存清空的触发点精确到行，大大降低缓存误杀率。参考资料

Mq管理平台 (mqcloud)

统一管理所有的消息队列，底层采用的是kafka（metaq？似乎在wiki中也看到metaq）

mq的应用场景
常见中间件：redis、rocketmq、rabbitmq、kafka，activemq等

  1. 解耦

降低业务系统之间的耦合，保证业务之间的逻辑不会混乱
例如：新用户注册发券场景
不使用mq的逻辑：用户在CRM注册，CRM调用营销系统的发券服务。当营销系统宕机后，CRM的会员注册功能将受到影响。
使用MQ的逻辑：用户在CRM注册，CRM发送会员注册的消息到MQ，完成会员的注册；营销系统订阅MQ，收到发券请求后，向指定的会员发券。营销系统宕机后，消息会保存在MQ中，等营销系统再次可用时，继续向用户发券。

  1. 异步处理

将非同步的任务放到mq中，比如发送短信邮件，推送日志数据等

  1. 流量削峰

比如秒杀的场景，利用队列进行排队，慢慢的进行消费，保证应用只处理可以应对的流量，其他的流量进行丢弃或者直接返回等操作，将流量尽可能的屏蔽在上层

  1. 数据同步

基于事件通知的方式，通过mq的消息持久化以及消息的事务等机制（最终一致性）

推送平台(NetPlatform)

NetPlatform 计划设计成一个统一的手机端长连接对外平台,即提供一个手机对外消息长连接服务的一个控制平台,后期可以灵活的添加新的长连接业务节点并且通过netManager进行统一调度.保证在不影响已有节点功能的基础上添加新的功能,并进行sdk版本控制等

数据分析平台(NetPlatform)

基于Spark的高速数据仓库，作为架构监控的数据仓库，也可以作为业务BI分析的数据仓库
数据平台使用hive对业务数据和hbase数据等进行管理，维护元数据信息，并能通过Spark SQL框架使用类似SQL的查询语言对相关数据进行分析和查询。Hive在Spark SQL中的主要作用就是负责为Spark SQL提供表与字段等相关的元数据服务，以及提供表在HDFS中的实际存储位置信息。
在数据平台中，主要通过三种方式为用户提供Spark相关功能的使用。
（1）图形化Spark SQL查询
（2）提交Jar作业以及python脚本
（3）通过客户端执行Spark SQL查询以及Spark作业提交
数据平台每天将来自关系型数据库以及hbase的业务数据同步至HDFS，然后通过Spark SQL为用户提供对这些业务数据的查询以及分析功能。同时也允许用户自定义相应的数据结构，然后利用Spark SQL存储和分析相关数据

神庙

神庙项目负责对业务部分关注的销售、推广信息进行收集、处理、分析。以hadoop为处理平台，以多样图表为展示门面。提供分析后的数据，为业务部门提供决策数据。（建立用户画像并针对其营销等）

波塞冬平台(POSEIDON)

波塞冬平台是一个用于管理，维护，监控的一套完整的实时计算管理平台。用户可以使用平台。提交 基于项目的实时任务。 也可以通过写CQL语句的方式提交任务。

public class DataSourceTransactionManager extends AbstractPlatformTransactionManager
        implements ResourceTransactionManager, InitializingBean {

    protected Object doGetTransaction() {
        DataSourceTransactionObject txObject = new DataSourceTransactionObject();
        txObject.setSavepointAllowed(isNestedTransactionAllowed());
        ConnectionHolder conHolder =
            (ConnectionHolder) TransactionSynchronizationManager.getResource(this.dataSource);
        txObject.setConnectionHolder(conHolder, false);
        return txObject;
    }
    
    //开始事务
    protected void doBegin(Object transaction, TransactionDefinition definition) {
        DataSourceTransactionObject txObject = (DataSourceTransactionObject) transaction;
        Connection con = null;

        try {
            if (txObject.getConnectionHolder() == null ||
                    txObject.getConnectionHolder().isSynchronizedWithTransaction()) {
                if(this.dataSource instanceof SddlLogicCluster){
                    
                    Map<Integer, SddlDataSource> map = ((SddlLogicCluster) this.dataSource).getSddlClusters().get(0).getMapDataSource();
                    Set<Map.Entry<Integer, SddlDataSource>> set = map.entrySet();
                    Iterator<Map.Entry<Integer, SddlDataSource>> ite = set.iterator() ;
                    //初始化 sharding connection
                    ShardingConnection newCon = new ShardingConnection();
                    while(ite.hasNext()){
                        Map.Entry<Integer, SddlDataSource> entry = ite.next();
                        Integer realDsIndex = entry.getKey(); // 库索引
                        Connection connection = entry.getValue().getDs().getConnection();
                        newCon.addShardingDs(connection, realDsIndex);
                    }
                    txObject.setConnectionHolder(new ConnectionHolder(newCon) , true);
                    
                }else{
                    Connection newCon = this.dataSource.getConnection();
                    if (logger.isDebugEnabled()) {
                        logger.debug("Acquired Connection [" + newCon + "] for JDBC transaction");
                    }
                    txObject.setConnectionHolder(new ConnectionHolder(newCon), true);
                }
                
                
            }

            txObject.getConnectionHolder().setSynchronizedWithTransaction(true);
            con = txObject.getConnectionHolder().getConnection();

            Integer previousIsolationLevel = DataSourceUtils.prepareConnectionForTransaction(con, definition);
            txObject.setPreviousIsolationLevel(previousIsolationLevel);

            // Switch to manual commit if necessary. This is very expensive in some JDBC drivers,
            // so we don't want to do it unnecessarily (for example if we've explicitly
            // configured the connection pool to set it already).
            if (con.getAutoCommit()) {
                txObject.setMustRestoreAutoCommit(true);
                if (logger.isDebugEnabled()) {
                    logger.debug("Switching JDBC Connection [" + con + "] to manual commit");
                }
                con.setAutoCommit(false);
            }
            txObject.getConnectionHolder().setTransactionActive(true);

            int timeout = determineTimeout(definition);
            if (timeout != TransactionDefinition.TIMEOUT_DEFAULT) {
                txObject.getConnectionHolder().setTimeoutInSeconds(timeout);
            }

            // Bind the session holder to the thread.
            if (txObject.isNewConnectionHolder()) {
                TransactionSynchronizationManager.bindResource(getDataSource(), txObject.getConnectionHolder());
            }
        }

        catch (Exception ex) {

            ConnDiscard.handleConnection(ex, con);

            DataSourceUtils.releaseConnection(con, this.dataSource);
            throw new CannotCreateTransactionException("Could not open JDBC Connection for transaction", ex);
        }
    }
    
    private static class DataSourceTransactionObject extends JdbcTransactionObjectSupport {

        private boolean newConnectionHolder;

        private boolean mustRestoreAutoCommit;

        public void setConnectionHolder(ConnectionHolder connectionHolder, boolean newConnectionHolder) {
            super.setConnectionHolder(connectionHolder);
            this.newConnectionHolder = newConnectionHolder;
        }

        public boolean isNewConnectionHolder() {
            return this.newConnectionHolder;
        }

        public boolean hasTransaction() {
            return (getConnectionHolder() != null && getConnectionHolder().isTransactionActive());
        }

        public void setMustRestoreAutoCommit(boolean mustRestoreAutoCommit) {
            this.mustRestoreAutoCommit = mustRestoreAutoCommit;
        }

        public boolean isMustRestoreAutoCommit() {
            return this.mustRestoreAutoCommit;
        }

        public void setRollbackOnly() {
            getConnectionHolder().setRollbackOnly();
        }

        public boolean isRollbackOnly() {
            return getConnectionHolder().isRollbackOnly();
        }
    }
}



/**
 *  sharding connection
 *  事物context下适配分库的所有connection
 * <br/> Created on 2015-12-4 下午12:15:16
 * @author  李洪波([email protected])
 * @since 4.1
 */
public class ShardingConnection implements Connection {
    public void commit() throws SQLException {
        Set<Map.Entry<Integer , Connection>> set = shardingMap.entrySet();
        Iterator<Map.Entry<Integer , Connection>> ite = set.iterator();
        while(ite.hasNext()){
            Map.Entry<Integer , Connection> entry = ite.next();
            Connection con = entry.getValue();
            if(con != null){
                con.commit();
            }
        }

    }
    
    public void rollback() throws SQLException {
        Set<Map.Entry<Integer , Connection>> set = shardingMap.entrySet();
        Iterator<Map.Entry<Integer , Connection>> ite = set.iterator();
        while(ite.hasNext()){
            Map.Entry<Integer , Connection> entry = ite.next();
            Connection con = entry.getValue();
            if(con != null){
                con.rollback();
            }
        }

    }
}
